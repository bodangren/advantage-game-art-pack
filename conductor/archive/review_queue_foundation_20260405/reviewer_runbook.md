# Reviewer Runbook

## Overview

The Review Queue provides a lightweight interface for human reviewers to approve, reject, or request changes to primitive candidates and compiled asset candidates generated by the asset pipeline.

## Accessing the Review Queue

```bash
# Start the review app
cd /path/to/project
pip install -e ".[review-app]"
REVIEW_DB=./review_queue.db python -m review_app.api

# Open in browser
open http://localhost:8000/queue
```

## Candidate Types

### Primitive Candidates
- Individual reusable parts (robes, weapons, props, tiles)
- Source: `primitive_library` program
- Review decision affects whether the primitive enters the part library

### Compiled Asset Candidates
- Full scene layouts, character sheets, or compositions
- Source: `scene_assembler` or other compiler programs
- Review decision affects whether the compiled asset is published

## Review Actions

| Action | Status | When to Use |
|--------|--------|-------------|
| **Approve** | `approved` | Candidate meets quality thresholds and style guide |
| **Reject** | `rejected` | Candidate has fundamental issues (bad palette, broken structure) |
| **Request Variant** | `needs_variant` | Candidate is promising but needs specific adjustments |
| **Promote** | `promoted` | Candidate is exceptional; fast-track to library |
| **Mark Gold** | `approved` | Candidate is an exemplary reference for future generation |

## Filtering the Queue

Use the filter controls to narrow down candidates:

- **Type**: Filter by `primitive` or `compiled_asset`
- **Status**: Focus on `pending` items to clear the queue
- **Theme**: Filter by `fantasy`, `scifi`, etc.
- **Min Confidence**: Show only high-scoring candidates
- **Sort**: Order by recency or confidence

## Variant Requests

When requesting a variant, provide a clear reason that the generator can act on:

**Good examples:**
- "Needs more gold trim on the sleeves"
- "Reduce shadow intensity by 20%"
- "Change primary color from red to blue"

**Poor examples:**
- "Make it better"
- "I don't like it"
- "Fix"

## Primitive Promotion

The `promote` action marks a primitive for priority inclusion in the part library. This bypasses the normal review queue for downstream compilation.

## Audit Trail

All decisions are recorded with timestamp and reason. This creates a complete audit trail for:
- Tracking reviewer patterns
- Understanding why specific variants were requested
- Auditing the approval process

## Integration Points

### Creating Candidates from Code

```python
from review_app.db import Database

db = Database("review_queue.db")
candidate_id = db.create_candidate(
    candidate_type="primitive",
    family="wizard_robe",
    source_program="primitive_library",
    rendered_files='["/path/to/robe.png"]',
    critic_scores='{"palette_score": 0.9}',
    confidence=0.85,
    theme="fantasy",
)
```

### Consuming Variant Requests

```python
# Find all candidates needing variants
needs_variant = db.list_candidates(status="needs_variant")
for candidate in needs_variant:
    decisions = db.get_decisions(candidate["id"])
    latest = decisions[-1]
    print(f"{candidate['family']}: {latest['reason']}")
```
